# LifeOS 设计方案

这份文档给 agent 和开发者看。目标是让后续接手者知道项目边界、数据结构、页面职责和不能乱改的约束。

## 产品定位

LifeOS 是一个本地优先的个人复盘系统。它不是团队协作工具，不是云端知识库，也不是多设备同步工具。

核心目标：

- 快速记录当天时间投入
- 把投入映射到“资产 / 维护 / 消耗”分类
- 管理项目和任务
- 生成每日复盘
- 通过仪表盘观察长期投入结构
- 允许用户自己导出和导入 JSON 备份

## 技术架构

```text
frontend/
  Vite + React 单页应用

backend/cloudflare-worker/
  可选 AI 解析服务扩展点，默认不参与本地启动

data/
  示例数据和说明，不保存真实用户数据

scripts/
  交付、演示数据、清理类脚本
```

默认运行路径：

```bash
./run.sh
```

`run.sh` 只启动前端，不启动后端。

## 前端入口

主要文件：

```text
frontend/src/main.jsx
frontend/src/styles.css
frontend/index.html
frontend/vite.config.js
```

当前 App 是单文件 React 主逻辑，状态、页面、事件处理都集中在 `frontend/src/main.jsx`。

后续如果拆分组件，必须保持数据兼容和页面行为稳定。

## 页面结构

页面由 `PAGES` 控制：

```js
["today", "projects", "tasks", "review", "dashboard", "assets", "backup"]
```

URL 使用 query 参数：

```text
?page=today
?page=projects
?page=tasks
?page=review
?page=dashboard
?page=assets
?page=backup
```

页面职责：

- `today`：今日快速记录
- `projects`：项目管理
- `tasks`：任务管理
- `review`：每日复盘卡和 PNG 导出
- `dashboard`：趋势和分类统计
- `assets`：资产分类定义
- `backup`：JSON 导入、导出、重置

不要把记录、复盘、仪表盘、资产定义重新堆回一个页面。

## 数据模型

浏览器存储 key：

```text
app:review-system:v1
```

顶层数据结构：

```json
{
  "projects": [],
  "tasks": [],
  "ideas": [],
  "events": [],
  "categories": [],
  "reviews": [],
  "schedules": [],
  "settings": {}
}
```

重要字段：

```json
{
  "settings": {
    "theme": "obsidian",
    "activeProjectId": null,
    "activePage": "today",
    "sidebarCollapsed": false,
    "dashboardView": "week",
    "lastOpenDate": "2026-05-20"
  }
}
```

核心数据实体：

- `projects`：长期项目
- `tasks`：项目任务
- `ideas`：想法池，当前 UI 中保留兼容
- `events`：时间记录，是统计和复盘的核心来源
- `categories`：资产分类
- `reviews`：每日复盘结果
- `schedules`：日程数据结构，保留兼容

## 兼容规则

必须保留：

- `STORAGE_KEY = "app:review-system:v1"`
- `createDefaultData()`
- `normalizeData(raw)`
- JSON 导入导出能力
- 旧数据自动补字段、自动升级的能力

任何新增字段都要通过 `normalizeData()` 做兼容处理。

不要直接假设 `localStorage` 里一定有完整字段。真实用户的数据可能来自旧版本，也可能缺字段。

## 核心业务规则

### 时间记录

`events` 是当天统计、仪表盘和复盘的数据基础。

每条记录至少需要：

```json
{
  "id": "event-id",
  "title": "英语阅读",
  "duration": 60,
  "category": "英语资产",
  "projectId": null,
  "taskId": null,
  "categoryId": null,
  "createdAt": "2026-05-20T10:00:00.000Z"
}
```

### 分类

分类有三种 `kind`：

- `asset`
- `maintenance`
- `drain`

分类的 `isPositive` 会影响正向资产时间和消耗时间统计。

### 项目和任务

任务通过 `projectId` 关联项目。项目进度由任务完成比例计算，不要手动写死。

如果 `settings.activeProjectId` 指向不存在的项目，`normalizeData()` 要自动修正为第一个可用项目或 `null`。

### 复盘

同一天保存复盘时，应替换当天旧复盘，而不是无限追加同一天多条复盘。

## 后端说明

`backend/cloudflare-worker/worker.js` 是可选的 AI 自然语言解析扩展点，使用 DeepSeek chat completions。

默认本地运行不依赖它。

如果要启用，需要单独部署 Cloudflare Worker，并设置 secret：

```bash
npx wrangler secret put DEEPSEEK_API_KEY
```

`.env.example` 只提供示例变量，不要提交真实 API key。

## 不要随便改的约束

- 不要默认引入登录系统
- 不要默认引入数据库
- 不要默认引入 Supabase 或云同步
- 不要把用户数据上传服务器
- 不要破坏 `localStorage` 旧数据兼容
- 不要移除 JSON 导入导出
- 不要把真实个人数据放进 `data/`
- 不要提交 `.env` 或 `.env.local`

如果要做云同步、AI 解析、账号系统或后端存储，必须作为明确的新需求处理。
